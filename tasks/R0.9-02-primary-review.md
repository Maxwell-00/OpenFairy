# BLOCKER 1 — Web operations are not bound to an immutable session

## Finding

`apps/web/app.js` keeps one mutable global:

```js
let selectedSession;
```

Recording finalization calls:

```js
await upload(wav);
```

`upload()` reads `selectedSession` once for the HTTP path and reads the global again after the await when it sends `voice.asr`.

Therefore this sequence is possible:

```text
record in session A
→ upload POST begins for session A
→ user selects/creates session B while POST is pending
→ upload returns artifact from A
→ voice.asr is sent with sid B
```

The input artifact is neutral and not session-owned, so the gateway can validly process that artifact in B. The audio has crossed session intent without an authorization failure.

The same issue exists earlier:

```text
start recording while A is selected
→ select B during recording
→ release
→ finalized audio uploads to B
```

The UI currently disables only the record button. `New session` and the session selector remain active during:

```text
recording
uploading
transcribing
thinking
synthesizing
```

## Subscription contamination

A socket is added to every attached session's subscriber set. Selecting a new session does not unsubscribe it from the old one.

The browser's `receive()` function accepts every projected event and ACK without checking:

- event SID;
- operation SID;
- selected-session generation;
- whether the response belongs to the current replay.

Consequences include:

- a late event from session A overwriting session B's transcript/final;
- a late TTS chunk from A triggering an MP3 fetch under B;
- a stale upload/fetch response replacing the active UI or playback;
- multiple historical TTS chunks during replay racing to replace the audio element.

## Required repair

Introduce an explicit browser-side session/operation generation.

At minimum:

1. Capture immutable values when a recording starts:

   ```text
   operationSid
   operationGeneration
   ```

2. Thread the same SID/generation through:

   ```text
   recorder finalization
   → upload endpoint
   → upload response
   → voice.asr send
   → ACK handling
   → TTS audio fetch
   → playback replacement
   ```

3. Never reread mutable `selectedSession` to complete an already-started operation.

4. Disable session creation and session selection while any operation is active.

   Acceptable active states:

   ```text
   recording
   uploading
   transcribing
   thinking
   synthesizing
   ```

5. A session switch from an idle state must:

   - increment the generation;
   - clear the prior projected view;
   - stop/revoke prior playback;
   - attach the selected session;
   - ignore all later frames from previous SIDs/generations.

6. Filter projected events and ACKs by SID.

   `session.created` may establish a new selected SID only when it corresponds to the current create-session action.

7. Bind `loadSpeech()` to explicit:

   ```text
   sid
   artifactId
   generation
   ```

   Before replacing playback, verify the same session/generation still owns the UI.

8. During `session.attach` replay:

   - consume canonical events;
   - retain the latest replayed TTS `audio_ref`;
   - wait for `session.resumed`;
   - fetch/play only the final selected replay audio;
   - do not launch one fetch/autoplay attempt per historical TTS chunk.

Do not change default socket subscription semantics merely to hide the client bug. A Web-specific current-session subscription replacement is acceptable only if it remains narrow and does not alter the default CLI/developer surface.

## Deterministic tests

Keep exactly the existing 14 primary case names. Strengthen existing cases and Web unit tests to prove:

### A. Switch during recording

```text
start recording in A
→ attempt session switch
→ switch is blocked or recording is deterministically discarded
→ no upload/ASR can be redirected to B
```

### B. Switch during upload

Use a deterministic deferred upload response:

```text
POST is bound to A
→ select/switch attempt
→ response resolves
→ voice.asr uses A or is discarded as stale
→ never B
```

### C. Late old-session event

```text
select A
→ select B
→ receive late speech.asr.final / turn.final / speech.tts.chunk for A
→ B view and playback remain unchanged
```

### D. Stale MP3 response

```text
start owned MP3 fetch for A
→ select B
→ A fetch resolves
→ it cannot replace B playback
```

### E. Multi-turn replay

```text
attach one session with multiple historical speech.tts.chunk events
→ replay completes
→ only latest audio_ref is fetched once
```

# BLOCKER 2 — Failure handling is silent or permanently unrecoverable

## Finding

The required state machine defines:

```text
any active state -> failed
failed -> ready only after local cleanup/reset
```

The implementation instead uses an internal state named:

```text
error
```

and provides no Reset/Retry control.

When upload or microphone handling calls:

```js
renderState("error", ...)
```

the record button becomes disabled, but:

- the socket remains connected;
- the session remains selected;
- no reset action exists;
- reconnect controls remain hidden;
- selecting a session does not restore `uiState`;
- the user must reload the page.

That does not satisfy the bounded failure/recovery contract.

## ACK errors are ignored

For any `voice.asr` ACK without `tts_audio_ref`, `receive()` currently returns the UI to `ready` without checking `error_status`.

This silently swallows:

- ASR route denial;
- ASR input failure;
- ASR cancellation;
- generic request failure;
- TTS failure after a successful text turn.

For TTS failure, the visible assistant text correctly survives, but the UI presents no bounded failure indicating that speech playback failed.

## Required repair

1. Use the required user-visible state name:

   ```text
   failed
   ```

2. Add a local Reset/Retry control.

3. Reset must:

   - invalidate the active generation;
   - discard/clean any recorder;
   - stop and revoke playback;
   - clear pending operation state;
   - preserve the selected canonical session;
   - return to `ready`;
   - send no cancel/interrupt/canonical event.

4. Handle `voice.asr` ACK status before treating it as success.

   Required behavior:

   - `none` with expected counts: normal success;
   - `asr_route_denied`: bounded failed state;
   - `asr_input_invalid`: bounded failed state;
   - `asr_cancelled` / cancelled: bounded failed state;
   - `request_failed`: bounded failed state;
   - TTS failure with a valid assistant final: preserve transcript/final, show bounded speech-playback failure, allow Reset/Retry.

5. Handle projected `op-error` and relevant bounded `progress.update` consistently.

6. Do not display provider IDs, routes, request IDs, raw error envelopes, paths, or credentials.

7. Add the missing minimum UI elements required by the brief:

   - visible guidance that a turn should normally be at most 30 seconds;
   - an explicit reload/replay session link for the selected hash session;
   - a Reset/Retry control.

8. Successful session creation/attach may restore `ready` only after the local stale operation state has been cleared.

## Required tests

Keep the 14 named cases. Extend Web unit tests and Cases H/J/M to prove:

- ASR route denial produces `failed`, not silent `ready`;
- TTS failure preserves assistant text and produces a bounded failure;
- upload failure produces `failed`;
- `op-error` produces `failed`;
- Reset performs local cleanup and returns to `ready`;
- Reset sends no cancel/interruption operation;
- the 30-second recommendation and replay/reset controls exist in the served page;
- no provider/private detail is rendered in any failure.

# BLOCKER 3 — Concurrent-upload test uses timing as synchronization

## Finding

Primary Case E starts one partial upload and then waits:

```ts
await new Promise((resolvePromise) => setTimeout(resolvePromise, 20));
```

before issuing the second upload.

The test assumes that 20 ms is enough for the server to:

- authenticate;
- validate the session;
- enter the upload reservation;
- install the request listeners.

That assumption is scheduler- and platform-dependent. It is exactly the kind of Windows CI timing dependency that the gated task required deterministic injected barriers to avoid.

A timeout may remain as a failure bound, but a sleep cannot be the primary proof that the critical state was reached.

## Required repair

Add a narrow constructor-only Web HTTP test seam, or an equivalent deterministic protocol signal.

Recommended shape:

```ts
interface WebVoiceHttpTestOptions {
  readonly afterUploadReservation?: (sid: string) => Promise<void>;
}
```

Rules:

- accepted only under `NODE_ENV=test` or `CI=true`;
- rejected in production otherwise;
- not available through YAML, CLI, environment, URL/query, or browser input;
- no production duration/body-limit override;
- bounded repository-owned behavior only.

Deterministic test sequence:

```text
first upload passes pre-body gates
→ first upload owns same-session reservation
→ afterUploadReservation barrier reached
→ second upload is issued
→ second upload returns 409
→ release first barrier
→ first upload completes 201
→ one artifact total
```

Remove the 20 ms sleep as synchronization.

Retain ordinary test deadlines only to fail a hung test.

## 2. Accepted high-risk surfaces

### Token/auth boundary — accepted

The static page is loopback-only, the token is kept in a module variable, the password input is cleared, HTTP uses Bearer auth, and WebSocket authentication remains the existing query-token connect path. Invalid WebSocket auth occurs before surface registration and message handling.

No gateway request logger currently records the WebSocket query string.

### Upload/WAV boundary — accepted subject to deterministic-test repair

The implementation provides:

- token/session/voice/busy/content gates before body reading;
- a bounded reader capped at `1,920,044` bytes;
- exact 44-byte PCM WAV parsing;
- 16 kHz/mono/PCM16 enforcement;
- exact size/sample/duration checks;
- gateway-derived labels/region;
- post-registration collision and content validation;
- zero provider/turn activity for over-limit input.

### MP3 ownership boundary — accepted

MP3 retrieval requires:

- valid token;
- valid existing session;
- canonical `speech.tts.chunk.audio_ref` ownership;
- exact artifact record;
- `kind:speech`;
- `audio/mpeg`;
- contained non-symlink regular file;
- stable open/read identity;
- exact size/hash/magic.

Existence and ownership failures collapse to `404`, and retrieval does not mutate JSONL.

### Provider projection — accepted

`surface=web-v0` allows only:

```text
session.create
session.attach
voice.asr
```

Outgoing canonical events and ACKs are projected through field allowlists. Default developer/CLI frames remain unchanged.

### ASR → turn → TTS architecture — accepted

The implementation preserves:

```text
one voice.asr path
one #submitVoiceFinalTranscript
one #acceptTurnInput
one TurnRunner
one coordinator-owned MiMo path
one coordinator-owned MiniMax path
```

The shared visible-final helper removes duplicated canonical TTS emission.

## 4. NIT

### N1 — `turn.delta` projection currently produces empty text

The generic turn projection extracts text from `payload.content`. Existing `turn.delta` payloads use a direct `text` field.

The minimal UI relies on `turn.final`, so this is not a closure blocker. Either project the direct delta text correctly or omit `turn.delta` from the Web projection to avoid sending empty updates.

### N2 — Stop transitions to `playback-ready` after removing the source

Local Stop revokes the Blob URL and removes `src`, then labels the state `playback-ready`. There is no remaining source to play.

Prefer `ready`/`stopped`, or retain a valid source if `playback-ready` is meant to support replay. Do not weaken the local-only stop rule.

### N3 — `server.ts` line ceiling has one line of headroom

The work report records `1628` non-empty lines against a `<1629` assertion. Several declarations/statements are compressed onto one physical line.

No provider orchestration was moved back into the server, so this is not a current blocker. Do not continue satisfying the architectural ceiling by formatting compression; future additions must extract responsibility into owned modules.
