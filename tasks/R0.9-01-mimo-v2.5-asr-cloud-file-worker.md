## 1. Objective

Implement the first real **cloud ASR** provider as a non-streaming, artifact-backed Xiaomi MiMo-V2.5-ASR worker.

R0.9-01 must prove that Fairy can:

- register an existing WAV or MP3 as a governed `input` artifact;
- route that labeled audio artifact through `speech.roles.asr`;
- deny an under-cleared cloud ASR before worker spawn, staging, connection, or request bytes;
- stage an allowed artifact into a private repository-external worker root;
- call MiMo-V2.5-ASR using a repository-owned stdlib-only Python worker;
- keep base64 audio strictly inside the worker/provider HTTP boundary;
- map one valid provider response to one canonical `speech.asr.final`;
- enter the existing voice-to-turn path exactly once;
- preserve `audio_ref`, labels, replay, cancellation, and cleanup;
- keep normal CI keyless through a provider-shaped local fake.

The accepted path is:

```text
owner/test WAV or MP3
  -> packages/artifacts as kind=input
  -> stable audio_ref
  -> authenticated gateway op using audio_ref only
  -> gateway-owned SpeechProviderCoordinator
  -> derive effective audio/voice labels
  -> speech.roles.asr candidate resolution
  -> shared provider clearance + speech egress
  -> private repository-external input staging
  -> supervised mimo_asr_worker.py
  -> worker-local base64 encoding
  -> MiMo pay-as-you-go HTTPS request
  -> provider-neutral asr.final worker result
  -> one canonical speech.asr.final
  -> existing #submitVoiceFinalTranscript
  -> existing #acceptTurnInput
  -> one existing TurnRunner
```

This task does **not** implement browser recording, Web UI, WebM, transcoding, streaming audio, real ASR partials, VAD, endpointing, local ASR, cloud fallback, MiMo TTS, or provider benchmarking.

## 2. Closed provider contract

R0.9-01 supports exactly one production ASR profile:

```text
provider family: Xiaomi MiMo
billing mode: pay-as-you-go
model: mimo-v2.5-asr
transport: mimo-v2.5-asr-chat-http
endpoint profile: mimo-paygo-cn
full endpoint: https://api.xiaomimimo.com/v1/chat/completions
authentication header: api-key
accepted credential kind: normal pay-as-you-go API key (sk-*)
language: auto | zh | en
request mode: non-streaming
input MIME: audio/wav | audio/mpeg
provider region: cn
```

Official MiMo documentation currently distinguishes:

```text
pay-as-you-go:
  base URL: https://api.xiaomimimo.com/v1
  key kind: sk-*

Token Plan:
  base URL: https://token-plan-cn.xiaomimimo.com/v1
  key kind: tp-*
```

R0.9-01 supports **pay-as-you-go only**.

Reject:

- Token Plan endpoint;
- Token Plan credentials;
- mixed key/endpoint modes;
- arbitrary endpoint URL;
- arbitrary model name;
- Bearer authentication;
- streaming response mode;
- Responses API or Anthropic envelope;
- a generic OpenAI SDK path.

### 2.1 Credential-kind validation

The gateway/worker must not log or persist the credential.

A bounded validation may establish only:

```text
credential_kind: paygo
```

The implementation may validate that the resolved credential has the expected pay-as-you-go key shape, but:

- never print the key;
- never print its prefix plus suffix;
- never hash it into diagnostics;
- never persist it in evidence;
- never send it to any endpoint other than the closed MiMo profile.

Named configuration/runtime failures:

```text
MIMO_ASR_CREDENTIAL_MISSING
MIMO_ASR_CREDENTIAL_KIND_MISMATCH
MIMO_ASR_ENDPOINT_PROFILE_UNSUPPORTED
```

### 2.2 Pricing and rate facts

MiMo-V2.5-ASR is currently billed by input audio duration for pay-as-you-go use. Rate limits are provider policy, not Fairy defaults.

R0.9-01 must classify provider `429` as retryable evidence, but must not automatically retry the current audio request in this slice.

## 8. Input artifact boundary

### 8.1 Accepted artifact contract

R0.9-01 accepts only:

```text
kind: input
mime: audio/wav | audio/mpeg
```

Add minimal WAV support to the neutral artifact package:

- `.wav` extension mapping;
- `audio/wav` MIME detection;
- no WebM;
- no Opus;
- no transcoder;
- no FFmpeg dependency;
- no registry rewrite.

Preserve existing MP3, perception, and speech artifact behavior.

A TTS artifact with `kind:speech` must be explicitly re-imported as `kind:input` before ASR. Do not silently cross artifact kinds.

### 8.2 Magic validation

Before staging/provider I/O:

For WAV:

- regular file;
- minimum header length;
- bytes `RIFF` at offset 0;
- bytes `WAVE` at offset 8;
- file size and registry size agree;
- SHA-256 agrees.

For MP3:

- regular file;
- accept existing supported MP3 magic policy:
  - `ID3`; or
  - valid MPEG frame sync;
- file size and registry size agree;
- SHA-256 agrees.

Reject:

- directories;
- symlinks;
- non-regular files;
- path escape;
- missing file;
- MIME/magic mismatch;
- size/hash mutation;
- unsupported kind;
- unsupported MIME;
- empty audio;
- input over hard raw-byte cap.

Do not parse full media duration or implement an audio decoder in this slice.

### 8.3 Conservative size cap

Use a hard raw-audio cap:

```text
7,000,000 bytes
```

This is deliberately below the provider's approximately 10 MB base64/request boundary.

Before reading/encoding provider body, compute the predicted base64 length:

```text
4 * ceil(raw_bytes / 3)
```

Then include the exact data-URL prefix and fixed JSON-envelope overhead in a bounded request-size check.

If the encoded body would exceed the closed provider cap, reject before provider connection.

Do not rely only on the raw-size check.

### 8.4 Effective labels

Derive effective labels from:

- input artifact sensitivity/residency;
- profile voice floor;
- authenticated request/frame labels;
- existing content escalation where applicable.

Rules:

- sensitivity uses raise-only maximum;
- residency uses existing governance composition;
- regions follow existing route law;
- request labels cannot lower artifact or voice-floor labels;
- the same effective labels are used for:
  - ASR provider clearance;
  - canonical `speech.asr.final`;
  - resulting `turn.input`;
  - downstream model route.

## 9. Private input staging

Audio bytes must not cross NDJSON.

For each allowed ASR request:

1. resolve `audio_ref`;
2. verify artifact kind/MIME/path/regular-file/size/hash/magic;
3. derive effective labels;
4. evaluate provider clearance and speech egress;
5. create a private repository-external temp root;
6. copy the verified audio to one fixed filename:
   - `asr-input.bin`;
7. re-check staged size and SHA-256;
8. launch the worker with the input root in a narrow child environment;
9. send metadata only over NDJSON;
10. remove the full root after success/failure/cancel/shutdown.

Proposed environment variable:

```text
FAIRY_SPEECH_WORKER_INPUT_ROOT
```

It is repository-derived and not user-configurable.

The wire may carry:

```text
input_token: asr-input.bin
```

No other value is accepted.

Reject:

- absolute path;
- `..`;
- nested path;
- drive-relative path;
- UNC path;
- alternate data stream/colon path;
- different input filename;
- path symlink;
- changed file between validation and staging;
- changed staged file before worker read.

The worker must never receive the artifact-store absolute path.

## 15. Provider response validation

A valid success requires:

- HTTP 2xx;
- response size within cap;
- JSON object;
- `object == "chat.completion"`;
- `model == "mimo-v2.5-asr"`;
- non-empty `id`;
- `choices` array;
- exactly one accepted choice at index `0`;
- `finish_reason == "stop"`;
- `message.role == "assistant"`;
- `message.content` non-empty string;
- transcript length within configured cap;
- no accepted tool call;
- no accepted audio output;
- no duplicate final.

Normalize transcript only by:

- Unicode-preserving string handling;
- trimming leading/trailing whitespace;
- normalizing CRLF/newline runs according to one deterministic documented rule.

Do not:

- translate;
- add punctuation through another model;
- correct spelling through another model;
- lowercase;
- rewrite Chinese;
- expose reasoning;
- accept `reasoning_content` as transcript;
- concatenate multiple choices.

Optional bounded evidence:

- provider response ID;
- usage seconds when valid and nonnegative;
- model;
- finish reason.

Do not persist the full usage object.

### 15.1 Empty/malformed response

Reject:

- empty/whitespace transcript;
- wrong model;
- missing choice/message/content;
- non-string content;
- content above cap;
- `tool_calls` with non-null value;
- unexpected audio result;
- streaming chunk/SSE response;
- invalid JSON;
- response over cap;
- duplicate provider final.

## 16. Error classification

Map provider and transport failures into bounded provider-neutral categories.

| Condition | Category | Retryable evidence | Automatic retry |
|---|---|---:|---:|
| local input/config invalid | invalid_request | no | no |
| HTTP 400 | invalid_request | no | no |
| HTTP 401 | unauthorized | no | no |
| HTTP 402 | balance_exhausted | no | no |
| HTTP 403 | access_denied | no | no |
| HTTP 404 | endpoint_or_model | no | no |
| HTTP 421 | safety_blocked | no | no |
| HTTP 429 | rate_limited | yes | no |
| HTTP 500 | provider_transient | yes | no |
| HTTP 503 | provider_unavailable | yes | no |
| DNS/connect/TLS | transport_failure | yes | no |
| read/total timeout | timeout | yes | no |
| malformed success | provider_protocol | no | no |
| transcript invalid | provider_protocol | no | no |

Rules:

- no same-candidate retry;
- no provider fallback in production R0.9-01;
- retryable is evidence only;
- errors contain no key, base64, audio, raw body, or absolute path;
- no canonical vendor-specific error family.

Use existing `progress.update`, `route.denied`, and `error` event contracts where applicable.

## 19. Cancellation and lifecycle

### 19.1 Cancel before final

On cancel before accepted `asr.final`:

- terminate/close the provider operation;
- no canonical ASR final;
- no `turn.input`;
- no model request;
- no retry;
- no orphan process;
- no staging root;
- replayable session.

If blocking stdlib HTTP cannot be interrupted cleanly, the gateway may terminate the worker after the existing cancellation deadline.

This is provider-operation cancellation, not barge-in.

### 19.2 Worker reuse decision

R0.9-01 should prefer the simplest lifecycle consistent with existing M3-05 behavior.

Gate decision required:

- reuse one worker per provider; or
- spawn one bounded worker per ASR operation.

Whichever is selected must:

- be explicit;
- have bounded concurrent requests;
- cleanly shut down;
- not introduce a second loop;
- preserve future coordinator compatibility.

Do not build a warm local-model pool.

### 19.3 Shutdown

Gateway shutdown:

- cancels active ASR operations;
- closes provider connection;
- terminates child workers;
- removes all staging roots;
- flushes replayable canonical state;
- leaves text-only gateway operation available after ASR setup/config failure.
