- One production `TurnRunner`.
- Modes remain policies, not alternate loops.
- JSONL session events remain the source of truth.

- No provider special case enters kernel.
- No browser state becomes canonical truth.

### 4.2 Voice-to-turn

The only ASR-final-to-turn construction path remains:

```text

  -> #acceptTurnInput
  -> one existing TurnRunner
```

## 9. Browser authentication and session flow

### 9.1 Gateway token

The UI provides a local gateway-token input.

The token:

- is held in a module-local in-memory variable;
- is not written to HTML;
- is not placed in the URL hash;
- is not placed in `localStorage`;
- is not placed in `sessionStorage`;
- is not placed in IndexedDB;
- is not stored in cookies;
- is cleared on reload;
- is never rendered after connection;
- is never included in committed fixtures.

HTTP requests use:

```text
Authorization: Bearer <gateway-token>
```

The browser WebSocket API cannot set an Authorization header, so the existing authenticated-connect mechanism may use:

```text
?token=<url-encoded-gateway-token>&surface=web-v0
```

The gateway must not echo that URL or token into events, ACKs, diagnostics, or UI output.

No new unauthenticated WebSocket listener is allowed.

Invalid/missing token:

```text
HTTP: 401
WebSocket: close 4401
processed ops: 0
created session: 0
created artifact: 0
```

### 9.2 Sessions

The UI must support:

- fetch existing sessions through authenticated `GET /sessions`;
- create a session through the existing `session.create` op;
- open a session through the existing `session.attach` op;
- subscribe before sending voice input;
- display the selected session ID/title;
- store only the session ID in the URL hash.

Proposed route:

```text
/web/#/sessions/ses_...
```

On reload:

1. the token is re-entered;
2. the UI reconnects with `surface=web-v0`;
3. the session ID is read from the hash;
4. `session.attach` replays from canonical JSONL;
5. transcript, assistant final, and available TTS audio are reconstructed from projected canonical events.

No browser-maintained chat history is authoritative.

## 11. Browser recording contract

### 11.1 Input API

Use:

```text
navigator.mediaDevices.getUserMedia
AudioContext
AudioWorklet or gate-approved equivalent Web Audio PCM path
```

Do not use `MediaRecorder` WebM output.

Microphone constraints may request:

```text
channelCount: 1
echoCancellation: true
noiseSuppression: true
autoGainControl: true
```

Treat constraints as browser hints, not proof of output format.

Use the actual capture context/track settings and convert the completed PCM to the fixed upload format.

### 11.2 Fixed output

The uploaded file must be canonical:

```text
container: RIFF/WAVE
format: PCM integer
audio format code: 1
sample rate: 16000
channels: 1
bits per sample: 16
block align: 2
byte rate: 32000
fmt chunk bytes: 16
header bytes: 44
data bytes: sample_count * 2
```

No extra chunks are needed.

The browser encoder must produce:

```text
RIFF at offset 0
WAVE at offset 8
fmt  at offset 12
data at offset 36
PCM data at offset 44
RIFF size = total bytes - 8
data size = total bytes - 44
```

### 11.3 Resampling

The source capture rate may differ from 16 kHz.

Use:

- browser-native `OfflineAudioContext` resampling; or
- a gate-approved deterministic dependency-free resampler.

Do not assume the microphone hardware or live AudioContext is already 16 kHz.

The final target sample count is authoritative.

### 11.4 Duration state

Constants:

```text
recommended_samples = 16000 * 30 = 480000
warning_samples     = 16000 * 50 = 800000
countdown_samples   = 16000 * 55 = 880000
max_samples         = 16000 * 60 = 960000
max_wav_bytes       = 44 + max_samples * 2 = 1920044
```

The displayed wall timer is advisory.

The recorder must derive elapsed/remaining time from captured audio converted to target-sample equivalents, not from `setTimeout` alone.

At 50 seconds:

```text
show remaining-10-second warning
```

At 55 seconds:

```text
show visible final countdown
```

At 60 seconds:

```text
stop capture
truncate to at most 960000 target samples
finalize WAV
call the same send function used by manual release
```

Auto-stop is exactly-once.

A delayed browser timer must not allow additional samples beyond the hard cap.

### 11.5 Push-to-talk interaction

Primary interaction:

```text
pointerdown -> start
pointerup -> stop and send
pointercancel / lost pointer capture -> bounded stop/discard decision
```

Support keyboard Space press/release when focus is not inside an input control.

Rules:

- pointer capture prevents accidental loss when the pointer leaves the button;
- repeat keydown does not start another recorder;
- a second recording cannot start while recording/uploading/transcribing/thinking/synthesizing;
- manual release and auto-stop share one finalize/upload path;
- a zero-sample or effectively empty recording is discarded;
- page reload, `pagehide`, tab close, or recorder failure stops media tracks and discards the unfinished buffer;
- unfinished or partial WAV is never uploaded;
- microphone tracks and AudioContext/AudioWorklet resources are closed after every attempt.

Do not implement server cancellation or barge-in from button release. Button release only completes the current recording.

## 12. Authenticated raw-WAV upload

Add one bounded same-origin endpoint, proposed:

```text
POST /web/api/sessions/:sid/input-audio
Content-Type: audio/wav
Authorization: Bearer <gateway-token>
Body: raw complete WAV bytes
```

Do not use:

- multipart;
- JSON;
- base64;
- data URLs;
- arbitrary filename/path fields;
- client-supplied artifact ID;
- client-supplied hash;
- client-supplied labels;
- client-supplied region;
- provider fields.

### 12.1 Pre-body gates

Before artifact registration:

- validate gateway token;
- validate session ID syntax;
- require the session to exist;
- require voice to be enabled;
- reject if the session already has an active turn/ASR;
- reject if another upload is active for the session;
- validate `Content-Type`;
- reject content encoding;
- if `Content-Length` is present, validate it before buffering.

The endpoint must use a bounded streaming body reader.

Stop reading and reject when the accumulated body exceeds:

```text
1920044 bytes
```

Do not buffer unbounded input.

### 12.2 Canonical WAV parser

The gateway must independently parse the uploaded WAV.

Accept exactly the closed browser format:

```text
minimum/maximum body: 46 .. 1920044 bytes
RIFF/WAVE
canonical 44-byte PCM header
audio format code 1
mono
16000 Hz
16-bit
block align 2
byte rate 32000
data chunk at offset 36
data size equals body bytes - 44
even PCM data byte count
sample_count = data_bytes / 2
1 <= sample_count <= 960000
duration_ms = floor(sample_count * 1000 / 16000)
```

Reject:

- missing/truncated header;
- wrong RIFF/WAVE/fmt/data magic;
- inconsistent RIFF size;
- non-PCM;
- float WAV;
- compressed WAV;
- stereo;
- wrong sample rate;
- wrong bit depth;
- wrong block alignment;
- wrong byte rate;
- extra/trailing bytes;
- declared size mismatch;
- odd data length;
- empty audio;
- sample count over 960000;
- duration over 60 seconds;
- body over 1920044 bytes.

A declared huge sample count in a short body must reject.

Do not add a general WAV parser or decoder.

### 12.3 HTTP result

Proposed success:

```text
HTTP 201
```

Bounded JSON:

```json
{
  "artifact_id": "art_...",
  "mime": "audio/wav",
  "size_bytes": 123456,
  "sample_count": 61706,
  "duration_ms": 3856,
  "labels": {
    "sensitivity": "personal",
    "residency": "region-restricted"
  }
}
```

Do not return:

- path;
- source filename;
- registry path;
- hash unless the gate explicitly requires it;
- provider information;
- secret reference;
- gateway token.

Proposed bounded failures:

```text
400 malformed request/session/body metadata
401 unauthorized
404 unknown session
409 session/upload busy
413 request body too large
415 unsupported content type or WAV format
422 invalid sample/duration contract
503 voice disabled or required governance configuration unavailable
```

Error JSON contains only a fixed code and bounded generic message.

## 15. Authenticated session-owned MP3 fetch

Add one bounded endpoint, proposed:

```text
GET /web/api/sessions/:sid/speech/:artifact_id
Authorization: Bearer <gateway-token>
```

The browser fetches the complete MP3 with `fetch`, creates a Blob/object URL, and supplies that object URL to an `<audio>` element.

Do not expose an artifact filesystem path to the browser.

### 15.1 Ownership proof

The artifact is fetchable only when:

1. the token is valid;
2. the session exists;
3. canonical events for that session contain:

```text
speech.tts.chunk.payload.audio_ref == requested artifact_id
```

4. the artifact registry record exists;
5. record kind is `speech`;
6. MIME is `audio/mpeg`;
7. file is regular and within the artifact root;
8. file size/hash/magic still match the registry record.

Ownership is proven by the session's canonical event reference, not by changing neutral artifact identity or storing a session owner in the content-addressed record.

A different session ID must not fetch the artifact even when the caller knows the artifact ID.

### 15.2 Response

Success:

```text
HTTP 200
Content-Type: audio/mpeg
Content-Length: exact bytes
Cache-Control: no-store
X-Content-Type-Options: nosniff
Content-Disposition: inline
```

Return the complete MP3.

Range requests are not required. The browser uses authenticated `fetch` followed by a Blob URL.

Failure:

```text
401 unauthorized
404 unknown session, unknown artifact, not owned, wrong kind, wrong MIME,
    missing/mutated file, or invalid artifact reference
```

Use 404 for existence/ownership failures to avoid artifact enumeration.

Never return:

- registry metadata;
- labels;
- artifact path;
- directory listing;
- arbitrary file bytes;
- provider response;
- source MP3 filename.
